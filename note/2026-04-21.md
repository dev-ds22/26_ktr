## 결론

지금 보신 경고는 **단일 플러그인 문제**가 아니라, **Jenkins core 2.414.3 자체가 너무 오래되어 여러 차수의 보안 공지 대상에 한꺼번에 포함된 상태**라는 뜻입니다. 공식 다운로드 기준 현재 Jenkins는 **LTS 2.555.1**, weekly는 **2.559**이며, LTS 라인은 보안 수정 백포트를 포함합니다. 따라서 우선순위는 **1) core 업그레이드, 2) 플러그인 업그레이드, 3) 권한/설정 완화** 순서로 잡는 것이 맞습니다. ([Jenkins](https://www.jenkins.io/download/ "Download and deploy"))

## 왜 지금은 “플러그인만 업데이트”가 답이 아닌가

현재 최신 Git client 6.6.0만 봐도 **Jenkins 2.516.3 이상**을 요구합니다. 즉, 현재 core 2.414.3 상태에서는 보안 수정 플러그인으로 바로 못 올라가는 항목이 생길 수 있습니다. 그래서 사용자 화면의 `no fix available`는 “세상에 fix가 없다”가 아니라, **현재 코어/호환성 범위에서 바로 적용 가능한 fix가 없다는 상황**일 가능성이 큽니다. ([Jenkins Plugins](https://plugins.jenkins.io/git-client/releases/ "Git client | Jenkins plugin"))

## 실무적으로 가장 현실적인 한 문장

**지금은 “플러그인 몇 개만 갱신”이 아니라, Java 버전 확인 후 Jenkins core를 먼저 최소 2.541.3 이상으로 끌어올리고, 가능하면 Java 21 기반으로 2.555.1 LTS까지 마무리한 다음 플러그인 전체를 정리하는 방식이 가장 안전합니다.** ([Jenkins](https://www.jenkins.io/doc/book/platform-information/support-policy-java/ "Java Support Policy"))  


## 전제

현재 상태를 **Jenkins 2.414.3 + Java 11**로 두고 보면, 바로 기억해야 할 핵심은 3개입니다. **Java 21 지원은 LTS 2.426.1부터**, **Java 11을 지원하는 마지막 LTS는 2.462.3**, **최신 LTS 2.555.1은 Java 21 또는 25가 필요**합니다. 또 Java 요구사항은 controller뿐 아니라 **agent와 CLI 등 Jenkins 시스템 구성요소 전체**에 적용됩니다. 따라서 **현재 2.414.3을 Java 21로 먼저 재기동하면 안 되고**, 업그레이드 창에서 런타임 전환과 core 업그레이드를 같이 설계해야 합니다. ([Jenkins](https://www.jenkins.io/doc/book/platform-information/support-policy-java/ "Java Support Policy"))  
실무 기준으로는 아래 2가지 경로 중 하나를 고르면 됩니다. 다만 Jenkins는 여러 LTS를 건너뛰어 올릴 때 **중간 LTS upgrade guide들도 전부 확인**하라고 공식 문서가 권고합니다. 또한 Java 전환이 들어가면 **플러그인은 업그레이드 전/후 모두 한 번씩 정리**하는 것이 안전합니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/ "Jenkins LTS Upgrade Guide"))

|경로|설명|권장도|
|---|---|---|
|최단 경로|정지 → Java 21 설치/지정 → Jenkins 최신 LTS(2.555.1) 업그레이드 → 기동|운영 단순성 높음|
|안전 경로|정지 → 필요 시 2.462.3 브리지 검증 → Java 21 지정 → 2.555.1 업그레이드|대규모 플러그인/에이전트 환경에 더 안전|

## 공통 사전 점검

공식 문서는 Java 업그레이드 전에 **`JENKINS_HOME` 백업**, 백업본으로 테스트, 이후 운영 반영을 권장합니다. Linux 패키지 설치형은 현재 **systemd 기반**이므로, 서비스 설정은 원본 unit 파일을 직접 수정하지 말고 **`systemctl edit jenkins`** 로 override를 두는 방식이 정석입니다. Linux 패키지 저장소는 **2026 signing key**가 적용되었으므로, 오래된 repo/key를 그대로 쓰면 업그레이드가 막힐 수 있습니다. ([Jenkins](https://www.jenkins.io/doc/book/platform-information/upgrade-java-to-21/ "Upgrade to Java 21"))

```bash
# 공통 확인
java -version
systemctl cat jenkins
journalctl -u jenkins -n 100 --no-pager
# 백업 예시
sudo systemctl stop jenkins
sudo tar -czf /backup/jenkins_home_$(date +%F).tar.gz /var/lib/jenkins
```

추가로, **2.479.1부터는 controller와 agent 모두 Java 17 이상**이어야 하므로, agent JVM도 같이 올릴 계획을 반드시 포함해야 합니다. Java 17 전환 가이드도 **agent JVM은 controller와 같은 JVM 버전**이어야 한다고 명시합니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/2.479/ "Upgrading to Jenkins LTS 2.479.x"))

## 1) apt 방식(Debian/Ubuntu)

Debian/Ubuntu 패키지형은 공식 문서 기준으로 **Java를 먼저 설치**하고, Jenkins 패키지는 stable repo에서 관리하는 방식입니다. 또한 service는 systemd로 관리되며, 포트나 Java 지정도 `systemctl edit jenkins` 방식이 권장됩니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/linux/ "Linux"))

### 권장 절차

1. Jenkins 정지 및 `JENKINS_HOME` 백업
    
2. Java 21 설치
    
3. Jenkins service가 Java 21을 사용하도록 지정
    
4. Jenkins apt stable repo와 2026 key 재등록
    
5. Jenkins 패키지 업그레이드
    
6. 기동 후 로그/플러그인/agent 확인  
    공식 Debian/Ubuntu 설치 문서와 systemd 문서를 기준으로 하면 아래 순서가 가장 실무적입니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/linux/ "Linux"))
    

```bash
# 1) 정지
sudo systemctl stop jenkins
# 2) Java 21 설치
sudo apt update
sudo apt install -y fontconfig openjdk-21-jre
java -version
# 3) Jenkins가 Java 21을 쓰도록 지정
sudo systemctl edit jenkins
```

`systemctl edit jenkins`가 열리면 아래처럼 넣습니다.

```ini
[Service]
Environment="JENKINS_JAVA_CMD=/usr/bin/java"
```

`/usr/bin/java`가 Java 21을 가리키게 하려면 필요 시 아래를 실행합니다.

```bash
sudo update-alternatives --config java
```

공식 문서는 Debian 계열에서 기본 Java를 바꾸려면 `update-alternatives --config java`를, 기본 Java를 바꾸기 싫으면 `JAVA_HOME` 또는 `JENKINS_JAVA_CMD`를 systemd override로 지정하라고 안내합니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/2.479/ "Upgrading to Jenkins LTS 2.479.x"))  
그다음 stable repo/key를 최신 기준으로 다시 넣고 업그레이드합니다. 2026 signing key 반영이 중요합니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/linux/ "Linux"))

```bash
sudo mkdir -p /etc/apt/keyrings
sudo wget -O /etc/apt/keyrings/jenkins-keyring.asc \
  https://pkg.jenkins.io/debian-stable/jenkins.io-2026.key
echo "deb [signed-by=/etc/apt/keyrings/jenkins-keyring.asc] https://pkg.jenkins.io/debian-stable binary/" | \
  sudo tee /etc/apt/sources.list.d/jenkins.list > /dev/null
sudo apt update
sudo apt install jenkins
sudo systemctl daemon-reload
sudo systemctl start jenkins
sudo systemctl status jenkins
journalctl -u jenkins -n 200 --no-pager
```

### apt 방식에서 특히 주의할 점

- **현재 2.414.3을 Java 21로 먼저 띄우지 말고**, 반드시 **정지 상태에서 Java 지정 + core 업그레이드**를 한 뒤 재기동하는 편이 안전합니다. Java 21 지원은 2.426.1 LTS부터이기 때문입니다. ([Jenkins](https://www.jenkins.io/doc/book/platform-information/support-policy-java/ "Java Support Policy"))
    
- 여러 LTS를 건너뛰는 업그레이드라면, 운영 반영 전 staging에서 한 번 올려보고 **plugin manager에서 전/후 2회 업데이트**하는 것이 좋습니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/ "Jenkins LTS Upgrade Guide"))
    

## 2) yum 방식(RHEL/CentOS/Rocky/AlmaLinux)

RHEL 계열도 Debian 계열과 동일하게 **systemd 기반**이며, 공식 Linux 문서는 stable rpm repo 사용과 Java 21 설치 예시를 제공합니다. Java 기본값을 바꾸려면 `alternatives --config java`, 바꾸기 싫으면 `systemctl edit jenkins`로 override를 두면 됩니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/linux/ "Linux"))

### 권장 절차

```bash
# 1) 정지
sudo systemctl stop jenkins
# 2) Java 21 설치
sudo yum install -y fontconfig java-21-openjdk
java -version
# 3) 필요 시 기본 Java 변경
sudo alternatives --config java
# 4) 또는 Jenkins에만 Java 지정
sudo systemctl edit jenkins
```

`systemctl edit jenkins`에는 아래처럼 넣으면 됩니다.

```ini
[Service]
Environment="JENKINS_JAVA_CMD=/usr/bin/java"
```

그다음 stable repo 기준으로 Jenkins를 올립니다. 공식 Linux 설치 문서의 RHEL 파트는 아래 흐름을 제시합니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/linux/ "Linux"))

```bash
sudo wget -O /etc/yum.repos.d/jenkins.repo \
  https://pkg.jenkins.io/rpm-stable/jenkins.repo
sudo yum upgrade
sudo yum install -y jenkins
sudo systemctl daemon-reload
sudo systemctl start jenkins
sudo systemctl status jenkins
journalctl -u jenkins -n 200 --no-pager
```

### yum 방식에서 특히 주의할 점

- 패키지형 Linux 설치는 **원본 service unit을 직접 수정하지 말고 override.conf 방식**을 쓰는 것이 공식 권장입니다. Debian은 `/lib/systemd/system/jenkins.service`, Red Hat 계열은 `/usr/lib/systemd/system/jenkins.service` 경로를 사용합니다. ([Jenkins](https://www.jenkins.io/doc/book/system-administration/systemd-services/ "Managing systemd services"))
    
- 2.541.1부터 Linux DEB/RPM 저장소 signing key가 바뀌었으므로, 오래된 repo 파일/키를 계속 쓰는 환경이면 repo 갱신이 필요합니다. ([Jenkins](https://www.jenkins.io/changelog-stable/ "LTS Changelog"))
    

## 3) Docker 방식

Docker는 패키지형과 다르게 **호스트 Java 11이 거의 의미가 없습니다.** 공식 Jenkins Docker 이미지는 컨테이너 안에 Java가 포함되어 있고, 공식 문서는 **LTS용 `jenkins/jenkins` 이미지**를 권장하며 예시로 **`jenkins/jenkins:2.555.1-jdk21`** 를 사용합니다. 또 2.479.1부터 Java 11 Docker 이미지는 retired 되었습니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/docker/ "Docker"))

### 가장 단순한 절차

1. 현재 컨테이너가 쓰는 `JENKINS_HOME` 볼륨 백업
    
2. 새 이미지 pull
    
3. 기존 컨테이너 stop/remove
    
4. 같은 volume/port/env로 새 컨테이너 재생성  
    공식 Docker 문서 흐름을 기준으로 하면 아래처럼 가면 됩니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/docker/ "Docker"))
    

```bash
# 1) 현재 컨테이너/볼륨 확인
docker ps -a
docker volume ls
# 2) 최신 LTS + JDK21 이미지 받기
docker pull jenkins/jenkins:2.555.1-jdk21
# 3) 기존 컨테이너 중지/삭제
docker stop jenkins
docker rm jenkins
# 4) 동일한 볼륨으로 재기동
docker run --name jenkins \
  --restart=on-failure \
  --detach \
  --publish 8080:8080 \
  --publish 50000:50000 \
  --volume jenkins-data:/var/jenkins_home \
  jenkins/jenkins:2.555.1-jdk21
# 5) 로그 확인
docker logs jenkins
```

### Dockerfile 커스텀 이미지였다면

공식 문서 예시처럼 base image만 바꾸면 됩니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/docker/ "Docker"))

```dockerfile
FROM jenkins/jenkins:2.555.1-jdk21
```

### Docker 방식에서 특히 주의할 점

- agent도 Docker 이미지 기반이면 **agent 쪽 Java도 controller 요구사항과 맞춰야** 합니다. 2.479.1부터는 controller/agent 모두 Java 17 이상이어야 합니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/2.479/ "Upgrading to Jenkins LTS 2.479.x"))
    
- 애플리케이션 빌드에 Java 8/11이 필요해도, **Jenkins 자체(remoting/controller)** 는 Java 17+ 또는 Java 21로 돌리고, 빌드용 JDK는 별도로 두면 됩니다. 공식 문서가 그 운영 방식을 허용합니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/2.479/ "Upgrading to Jenkins LTS 2.479.x"))
    

## 4) WAR 방식

WAR 방식은 가장 단순합니다. 공식 문서는 Jenkins WAR 실행을 **`java -jar jenkins.war`** 로 설명합니다. 따라서 현재 서버 기본 Java가 11이어도, **Java 21을 별도 설치한 뒤 Jenkins 실행에만 그 Java 21 바이너리를 명시**하면 됩니다. ([Jenkins](https://www.jenkins.io/doc/book/installing/war-file/ "WAR file"))

### 권장 절차

1. 현재 Jenkins 프로세스 중지
    
2. `JENKINS_HOME` 백업
    
3. Java 21 설치
    
4. `jenkins.war` 교체
    
5. Java 21로 실행
    
6. 로그 확인
    

```bash
# 1) 중지 (예: systemd 사용 시)
sudo systemctl stop jenkins
# 2) 백업
sudo tar -czf /backup/jenkins_home_$(date +%F).tar.gz /var/lib/jenkins
cp /opt/jenkins/jenkins.war /opt/jenkins/jenkins.war.bak.$(date +%F)
# 3) Java 21 설치 후 경로 확인
java -version
# 4) 최신 WAR 다운로드
wget -O /opt/jenkins/jenkins.war \
  https://updates.jenkins.io/download/war/2.555.1/jenkins.war
# 5) Java 21로 명시 실행
/path/to/jdk-21/bin/java -jar /opt/jenkins/jenkins.war
```

WAR 방식은 install method가 단순한 대신, Java 경로와 실행 스크립트를 직접 관리해야 합니다. 다만 **여러 LTS를 건너뛸 때는 중간 upgrade guide 확인**, **plugin 사전/사후 업데이트**, **agent JVM 동기화**는 여전히 필요합니다. ([Jenkins](https://www.jenkins.io/doc/upgrade-guide/ "Jenkins LTS Upgrade Guide"))

## 업그레이드 직후 체크리스트

아래 5개는 설치 방식과 상관없이 공통입니다. 공식 문서가 plugin과 agent 호환성, systemd 로그 확인을 특히 강조합니다. ([Jenkins](https://www.jenkins.io/doc/book/platform-information/upgrade-java-to-17/ "Upgrade to Java 17"))

|체크 항목|확인 방법|
|---|---|
|Jenkins 기동 여부|`systemctl status jenkins` 또는 `docker logs`|
|플러그인 로드 오류|관리자 화면, `journalctl -u jenkins`, 컨테이너 로그|
|agent Java 버전|노드 접속 테스트, Versions Node Monitors|
|핵심 잡 실행|대표 Pipeline/Freestyle 1~2개 수동 실행|
