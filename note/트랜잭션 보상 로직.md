## 결론

현재 전제는 **`dao`는 Spring 트랜잭션 대상**, **`mDao`는 트랜잭션 누락 또는 AutoCommit 가능성 있음**입니다. 이 경우 `mDao`까지 하나의 원자적 트랜잭션으로 묶인 것이 아니므로, 실무적으로는 **롤백이 아니라 보상 트랜잭션**으로 처리해야 합니다. Spring은 명시적 트랜잭션 제어가 필요한 경우 `TransactionTemplate` 사용을 제공하며, 선언적 트랜잭션은 기본적으로 `RuntimeException`/`Error`에서 롤백됩니다. ([Home](https://docs.spring.io/spring-framework/reference/data-access/transaction/programmatic.html?utm_source=chatgpt.com "Programmatic Transaction Management"))

## 권장 처리 흐름

```text
1. dao 삭제 대상 백업 조회
2. dao 삭제를 main DB 트랜잭션으로 먼저 처리
3. dao 트랜잭션 commit 완료
4. mDao 삭제 처리
5. mDao 실패 시 dao 삭제분을 보상 트랜잭션으로 복원
6. 보상 실패 시 운영자 확인 가능한 오류 로그/복구 대상 저장
```

## 수정 예시

> 전제: `deleteTemplate()` 메소드 자체가 기존 AOP 트랜잭션으로 감싸져 있으면 `dao commit 후 mDao 처리`가 명확하지 않을 수 있습니다. 이 방식으로 가려면 `deleteTemplate()`은 외부 트랜잭션 대상에서 제외하거나, 아래처럼 `TransactionTemplate`을 `REQUIRES_NEW`로 분리하는 편이 안전합니다.

```java
import org.springframework.transaction.PlatformTransactionManager;
import org.springframework.transaction.TransactionDefinition;
import org.springframework.transaction.support.TransactionTemplate;
import java.util.*;
public class TemplateService {
    private final Dao dao;
    private final MailDao mDao;
    private final TransactionTemplate mainTxTemplate;
    public TemplateService(Dao dao, MailDao mDao, PlatformTransactionManager txManager) {
        this.dao = dao;
        this.mDao = mDao;
        this.mainTxTemplate = new TransactionTemplate(txManager);
        this.mainTxTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);
    }
    public void deleteTemplate(Map<String, Object> deleteMap, Map<String, Object> loginUserInfo) {
        Map<String, Object> param = new HashMap<>(deleteMap);
        param.put("UPDUSR_ID", loginUserInfo.get("USER_ID"));
        TemplateDeleteBackup mainBackup = null;
        MailDeleteBackup mailBackup = null;
        try {
            /*
             * 1. main DB 삭제 + 백업을 하나의 트랜잭션으로 처리
             * - 백업 조회는 삭제 직전에 수행
             * - 삭제 성공 시 commit
             * - 삭제 실패 시 자동 rollback
             */
            mainBackup = mainTxTemplate.execute(status -> {
                TemplateDeleteBackup backup = selectMainBackup(param);
                dao.deleteReplcInfo(param);
                dao.deleteRcvTgt(param);
                dao.deleteTemplate(param);
                return backup;
            });
            /*
             * 2. mDao 처리 전 백업
             * - mDao가 트랜잭션 대상이 아니면 deleteEmsReplcInfo 성공 후 deleteEmsTemp 실패 같은 부분 반영 위험이 있음
             * - 가능하면 mDao에도 별도 txManagerMail을 추가하는 것이 더 안전
             */
            mailBackup = selectMailBackup(param);
            /*
             * 3. mDao 삭제 처리
             * - 현재 전제상 mDao는 트랜잭션 보장 없음
             */
            mDao.deleteEmsReplcInfo(param);
            mDao.deleteEmsTemp(param);
        } catch (Exception e) {
            RuntimeException resultException =
                new TemplateDeleteFailException("템플릿 삭제 중 mDao 처리 실패 또는 보상 처리 대상 발생", e);
            /*
             * 4. mDao 일부 삭제 보상
             * - mDao 백업이 있는 경우에만 복원 시도
             * - mDao가 AutoCommit이면 이 복원도 보상 작업일 뿐 원자적 rollback은 아님
             */
            if (mailBackup != null) {
                try {
                    compensateMailDelete(mailBackup);
                } catch (Exception compensateMailEx) {
                    resultException.addSuppressed(compensateMailEx);
                }
            }
            /*
             * 5. main DB 삭제 보상
             * - dao 삭제가 commit된 이후 mDao가 실패했을 수 있으므로 main DB 복원
             */
            if (mainBackup != null) {
                try {
                    compensateMainDelete(mainBackup);
                } catch (Exception compensateMainEx) {
                    resultException.addSuppressed(compensateMainEx);
                }
            }
            throw resultException;
        }
    }
    private TemplateDeleteBackup selectMainBackup(Map<String, Object> param) {
        TemplateDeleteBackup backup = new TemplateDeleteBackup();
        backup.setTemplate(dao.selectTemplateForRestore(param));
        backup.setRcvTgtList(dao.selectRcvTgtForRestore(param));
        backup.setReplcInfoList(dao.selectReplcInfoForRestore(param));
        return backup;
    }
    private MailDeleteBackup selectMailBackup(Map<String, Object> param) {
        MailDeleteBackup backup = new MailDeleteBackup();
        backup.setEmsTemp(mDao.selectEmsTempForRestore(param));
        backup.setEmsReplcInfoList(mDao.selectEmsReplcInfoForRestore(param));
        return backup;
    }
    private void compensateMainDelete(TemplateDeleteBackup backup) {
        mainTxTemplate.executeWithoutResult(status -> {
            /*
             * 복원 순서 주의:
             * 부모 테이블 -> 자식 테이블 순서 권장
             */
            if (backup.getTemplate() != null) {
                dao.restoreTemplate(backup.getTemplate());
            }
            if (backup.getRcvTgtList() != null) {
                for (Map<String, Object> row : backup.getRcvTgtList()) {
                    dao.restoreRcvTgt(row);
                }
            }
            if (backup.getReplcInfoList() != null) {
                for (Map<String, Object> row : backup.getReplcInfoList()) {
                    dao.restoreReplcInfo(row);
                }
            }
        });
    }
    private void compensateMailDelete(MailDeleteBackup backup) {
        /*
         * mDao에 트랜잭션이 없다는 전제이므로 보상 복원도 부분 성공 가능성이 있음
         * 가능하면 txManagerMail을 추가해서 이 부분도 TransactionTemplate으로 감싸는 것이 좋음
         */
        if (backup.getEmsTemp() != null) {
            mDao.restoreEmsTemp(backup.getEmsTemp());
        }
        if (backup.getEmsReplcInfoList() != null) {
            for (Map<String, Object> row : backup.getEmsReplcInfoList()) {
                mDao.restoreEmsReplcInfo(row);
            }
        }
    }
}
```

## 실무 주의점

|구분|주의점|
|---|---|
|트랜잭션 의미|`mDao`에 트랜잭션이 없으면 DB 차원의 rollback이 아니라 수동 복원입니다.|
|복원 SQL|`INSERT`만 쓰면 중복키 오류가 날 수 있으므로 `UPSERT` 또는 존재 여부 확인 후 복원 방식이 안전합니다.|
|순서|삭제는 자식 → 부모, 복원은 부모 → 자식 순서가 일반적으로 안전합니다.|
|동시성|삭제 대상 백업 조회 시점과 삭제 시점 사이 변경 가능성이 있으면 `SELECT ... FOR UPDATE` 검토가 필요합니다.|
|예외 처리|보상 실패까지 로그에 남겨야 운영자가 수동 복구할 수 있습니다.|
|최선책|가능하면 `mDao`에도 `txManagerMail`을 추가하고 별도 트랜잭션으로 묶는 것이 더 안전합니다.|

## 더 안전한 개선안

`mDao`에도 트랜잭션 매니저를 추가할 수 있다면 아래 구조가 더 낫습니다.

```java
private final TransactionTemplate mailTxTemplate;
public TemplateService(Dao dao,
                       MailDao mDao,
                       PlatformTransactionManager txManager,
                       PlatformTransactionManager txManagerMail) {
    this.dao = dao;
    this.mDao = mDao;
    this.mainTxTemplate = new TransactionTemplate(txManager);
    this.mainTxTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);
    this.mailTxTemplate = new TransactionTemplate(txManagerMail);
    this.mailTxTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);
}
```

```java
mailTxTemplate.executeWithoutResult(status -> {
    mDao.deleteEmsReplcInfo(param);
    mDao.deleteEmsTemp(param);
});
```

이렇게 하면 `mDao.deleteEmsReplcInfo()` 성공 후 `mDao.deleteEmsTemp()` 실패 시 **mail DB 내부에서는 rollback 가능**합니다. 다만 `dao`와 `mDao`가 서로 다른 DB라면 여전히 전체 원자성은 보장되지 않으므로, `mDao` 트랜잭션 실패 시 `dao` 보상 로직은 유지해야 합니다.분산 트랜잭션 수준의 원자성이 필요하면 JTA/XA 같은 별도 설계를 검토해야 합니다.

## 간단 수정 방향

`dao` 처리 중 Exception이 발생하면 즉시 중단하고, `dao` 처리가 모두 정상 완료된 경우에만 `mDao`를 실행합니다. `mDao` 처리 중 Exception이 발생하면 이미 완료된 `dao` 처리에 대한 **보상 로직 개념**만 주석/메소드 형태로 추가합니다.

## 수정 코드

```java
public void deleteTemplate(Map<String, Object> deleteMap, Map<String, Object> loginUserInfo) {
    // 수정자 ID
    deleteMap.put("UPDUSR_ID", loginUserInfo.get("USER_ID"));
    boolean daoSuccess = false;
    try {
        /*
         * 1. Main DB 처리
         * dao는 트랜잭션 설정이 되어 있다는 전제
         * 여기서 Exception 발생 시 Spring Transaction에 의해 rollback 대상
         */
        dao.deleteReplcInfo(deleteMap);
        dao.deleteRcvTgt(deleteMap);
        dao.deleteTemplate(deleteMap);
        daoSuccess = true;
    } catch (Exception e) {
        /*
         * dao 처리 실패
         * - mDao 처리하지 않음
         * - 반드시 다시 throw 해야 Spring 트랜잭션 rollback 가능
         */
        throw e;
    }
    if (daoSuccess) {
        try {
            /*
             * 2. Mail DB 처리
             * mDao는 트랜잭션 설정이 누락되어 있다는 전제
             */
            mDao.deleteEmsReplcInfo(deleteMap);
            mDao.deleteEmsTemp(deleteMap);
        } catch (Exception e) {
            /*
             * 3. 보상 로직 개념
             * mDao 실패 시 dao는 이미 정상 처리되었을 수 있으므로
             * 삭제된 Main DB 데이터를 복구하거나,
             * 복구 대상 이력을 남겨 후속 배치/수동 복구 대상으로 처리
             */
            compensateDeleteTemplate(deleteMap, e);
            throw e;
        }
    }
}
```

## 중요한 주의점

|구분|설명|
|---|---|
|`catch` 후 `throw`|`dao` Exception을 잡고 다시 던지지 않으면 Spring 트랜잭션이 정상 종료로 판단해 commit될 수 있습니다.|
|`daoSuccess`|`dao`가 모두 정상 완료된 경우에만 `mDao`를 실행하기 위한 최소 안전장치입니다.|
|`mDao` 실패|`mDao`는 트랜잭션 누락 상태이므로 부분 반영 가능성이 있습니다. 예: `deleteEmsReplcInfo()` 성공 후 `deleteEmsTemp()` 실패|
|보상 로직|DB rollback이 아니라 업무적으로 되돌리는 복구 처리입니다.|
|정확한 보상|삭제 전 데이터를 미리 백업해두지 않으면 완전 복구가 어렵습니다.|

## 더 단순한 형태

`daoSuccess` 변수를 없애도 됩니다. `dao`에서 실패하면 `throw`로 중단되기 때문에 아래 코드도 동작 흐름은 같습니다.

```java
public void deleteTemplate(Map<String, Object> deleteMap, Map<String, Object> loginUserInfo) {
    deleteMap.put("UPDUSR_ID", loginUserInfo.get("USER_ID"));
    try {
        dao.deleteReplcInfo(deleteMap);
        dao.deleteRcvTgt(deleteMap);
        dao.deleteTemplate(deleteMap);
    } catch (Exception e) {
        throw e;
    }
    try {
        mDao.deleteEmsReplcInfo(deleteMap);
        mDao.deleteEmsTemp(deleteMap);
    } catch (Exception e) {
        compensateDeleteTemplate(deleteMap, e);
        throw e;
    }
}
```

## 실무 판단

현재 요청 기준으로는 **두 번째 코드가 가장 간단**합니다. 다만 `deleteTemplate()` 메소드 전체가 이미 `@Transactional` 또는 AOP 트랜잭션 대상이면, `mDao` 실패 후 `throw e`가 발생하면서 `dao` 트랜잭션도 rollback될 수 있습니다. 이 경우 별도 보상 로직은 실제로 실행되더라도 최종 rollback에 포함될 수 있으므로, 보상 로직을 실효성 있게 만들려면 `REQUIRES_NEW` 또는 별도 이력 저장 방식이 필요합니다.
