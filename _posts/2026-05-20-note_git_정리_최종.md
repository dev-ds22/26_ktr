## 5. push 전 dev 최신 변경 반영

```bash
# 5-1. dev 최신 내용 가져오기
git fetch origin 

# 5-2. feature 브랜치에서 dev 병합
git switch feature/user
git merge origin/dev
```

`feature/user`에서 `git merge origin/dev` 중 충돌이 났고, **충돌 파일은 일단 `origin/dev` 최신 파일 기준으로 가져온 뒤 내 작업 내용을 다시 추가**하려면 아래 방식으로 처리하면 됩니다.

```bash
git checkout --theirs -- 충돌파일\

# 또는 최신 Git

git restore --source=MERGE_HEAD -- 충돌파일
```

- 충돌 파일 내용이 origin/dev 기준 최신 파일로 바뀜
- 내 feature/user 쪽 변경 내용은 해당 파일에서는 일단 제거됨

### 충돌 발생 시

## 검토 결과

- 현재 정리한 흐름은 **실무에서 사용 가능**합니다. 다만 아래 5가지는 수정하는 것이 좋습니다.

|  항목 | 현재 내용                  | 개선 의견                                                                       |
| --: | ---------------------- | --------------------------------------------------------------------------- |
|   1 | `dev`로 이동 후 `pull`     | 신규 feature 생성만 목적이면 `origin/dev`에서 바로 생성하는 방식이 더 안전함                        |
|   2 | `git add .`            | 실무에서는 불필요 파일 포함 위험이 있어 `git status` 확인 후 선택 add 권장                          |
|   3 | `git merge origin/dev` | 맞음. 단, `fetch` 후 merge해야 함                                                  |
|   4 | 충돌 시 `--theirs`        | merge 기준에서는 맞음. 현재 브랜치가 `feature/user`이면 `--theirs`는 병합 대상인 `origin/dev` 쪽임 |
|   5 | `git branch -D`        | MR merge 확인 후에만 사용. 특히 Squash merge면 `-d`가 실패할 수 있음                         |

- `git pull --ff-only`는 로컬 브랜치가 원격 브랜치와 분기되지 않았을 때만 fast-forward로 갱신하고, 분기되어 있으면 실패하는 방식이므로 `dev` 최신화에는 안전한 선택입니다. ([Git](https://git-scm.com/docs/git-pull?utm_source=chatgpt.com "Git - git-pull Documentation"))
- Git merge 충돌이 발생하면 충돌 파일은 unmerged 상태가 되고, 해결 후 `git add` 또는 필요 시 `git rm`으로 해결 표시를 해야 합니다. ([Git](https://git-scm.com/docs/git-checkout/2.27.0?utm_source=chatgpt.com "Git - git-checkout Documentation"))
- GitLab MR의 `Delete source branch`는 MR 생성 시 또는 merge 시 선택할 수 있고, merge 후 source branch를 삭제하는 기능입니다. ([GitLab 문서](https://docs.gitlab.com/user/project/merge_requests/?utm_source=chatgpt.com "Merge requests"))
## 수정 권장 안

### 1. 원격 최신 정보 가져오기

```bash
git fetch --prune origin
```

`feature/user`를 새로 만들 목적이라면 굳이 로컬 `dev`로 이동해서 `pull`하지 않아도 됩니다. `origin/dev` 기준으로 바로 브랜치를 만들면 현재 원격 `dev` 기준이 명확합니다.

### 2. `origin/dev` 기준으로 신규 feature 브랜치 생성

```bash
git switch -c feature/user origin/dev
```

이 방식이 기존 방식보다 실무에서 더 깔끔합니다.  

기존 방식:

```bash
git switch dev
git pull --ff-only origin dev
git switch -c feature/user
```

개선 방식:

```bash
git fetch --prune origin
git switch -c feature/user origin/dev
```

둘 다 가능하지만, **신규 작업 브랜치 생성 목적이면 개선 방식이 더 단순하고 실수 가능성이 낮습니다.**

### 3. 작업 진행

```bash
# 소스 수정
```

작업 중간 확인:

```bash
git status
git diff
```

### 4. 작업 후 commit

`git add .`도 가능하지만, 실무에서는 먼저 변경 파일을 확인하는 것이 좋습니다.

```bash
git status
```

전체 add:

```bash
git add .
```

또는 필요한 파일만 add:

```bash
git add src/main/java/com/example/TestService.java
```

커밋:

```bash
git commit -m "작업 내용"
```

실무 개선점:

```bash
git add .
```

는 `.log`, 임시 파일, IDE 설정 파일 등이 같이 들어갈 수 있으므로 `.gitignore`가 잘 관리되어 있어야 합니다.

### 5. push 전 `dev` 최신 변경 반영

MR 전에 다른 사람이 `dev`에 반영한 최신 변경을 내 feature 브랜치에 반영합니다.

```bash
git fetch origin
git switch feature/user
git merge origin/dev
```

이 순서는 적절합니다.  
의미:

```text
현재 브랜치: feature/user
병합 대상: origin/dev
```

따라서 merge 충돌 상황에서:

```bash
git checkout --ours -- 파일명
```

은 `feature/user` 쪽 내용을 선택합니다.

```bash
git checkout --theirs -- 파일명
```

은 `origin/dev` 쪽 내용을 선택합니다.

## 충돌 발생 시 처리

### 5-1. 충돌 파일 확인

```bash
git status
```

### 5-2. 충돌 파일을 일단 `origin/dev` 기준으로 선택

```bash
git checkout --theirs -- src/main/java/com/example/TestService.java
```

또는 최신 Git에서는:

```bash
git restore --source=MERGE_HEAD -- src/main/java/com/example/TestService.java
```

현재 상황이 `feature/user`에서 `origin/dev`를 merge하는 경우라면, `MERGE_HEAD`는 병합 대상인 `origin/dev`를 가리키므로 위 명령은 `origin/dev` 쪽 파일을 가져오는 의미로 볼 수 있습니다.

### 5-3. 파일 열어서 내 작업 내용 다시 반영

```bash
# IDE 또는 편집기로 파일 수정
```

주의할 점:

```text
git checkout --theirs -- 파일명
```

을 실행하면 해당 충돌 파일에서 내 feature 브랜치의 변경 내용은 일단 사라집니다.  
따라서 반드시 내 작업 내용을 다시 추가해야 합니다.

### 5-4. 변경 내용 확인

```bash
git diff
```

### 5-5. 충돌 해결 표시

```bash
git add src/main/java/com/example/TestService.java
```

### 5-6. merge commit 생성

```bash
git commit
```

Git이 merge commit 메시지를 자동으로 열면 그대로 저장해도 됩니다.  
예:

```text
Merge remote-tracking branch 'origin/dev' into feature/user
```


## 9. MR 머지

GitLab 페이지에서 merge 진행 시:

```text
Delete source branch 선택
```

이 옵션은 **원격 branch인 `origin/feature/user` 삭제**에 해당합니다. 로컬 PC의 `feature/user` 브랜치는 자동 삭제되지 않습니다.

## 10. 원격 삭제 상태 반영

MR merge 후 로컬에서 원격 브랜치 삭제 상태를 반영합니다.

```bash
git fetch --prune origin
```

확인:

```bash
git branch -r
```

`origin/feature/user`가 보이지 않으면 원격 브랜치 삭제가 반영된 것입니다.

## 11. local dev 최신화

```bash
git switch dev

```

MR merge 결과가 `dev`에 반영되었는지 확인:

```bash
git log --oneline -n 10
```

## 12. local feature branch 삭제

일반 삭제:

```bash
git branch -d feature/user
```

강제 삭제:

```bash
git branch -D feature/user
```

`-d`는 Git이 해당 브랜치가 병합되었다고 판단할 때 삭제됩니다.  
`-D`는 강제 삭제이므로 MR이 정상 merge 되었고 더 이상 로컬 작업이 필요 없다는 것을 확인한 뒤에만 사용해야 합니다.  
특히 GitLab에서 **Squash merge**를 사용한 경우, 실제 변경은 `dev`에 반영되었더라도 커밋 이력이 달라서 `git branch -d feature/user`가 실패할 수 있습니다. 이 경우 MR merge 완료와 `dev` 반영을 확인한 뒤 `git branch -D feature/user`를 사용할 수 있습니다.

## 추가 주의 사항

### 1. `origin.dev`가 아니라 `origin/dev`

정확한 표현은:

```bash
origin/dev
```

입니다.

### 2. merge 전에 내 작업은 commit 되어 있는 것이 안전

좋은 흐름:

```text
작업 → commit → fetch → merge origin/dev → 충돌 해결 → test → push → MR
```

피해야 할 흐름:

```text
작업 중 미커밋 상태 → merge origin/dev
```

이 경우 충돌과 내 작업 변경이 섞여서 원복과 비교가 어려워질 수 있습니다.

### 3. 미완성 작업이면 stash 사용

커밋하기 애매한 상태에서 `dev` 최신 변경을 먼저 반영해야 한다면:

```bash
git stash push -u -m "feature/user 작업 임시저장"
git fetch origin
git merge origin/dev
git stash pop
```

이후 충돌이 나면 수정 후:

```bash
git add .
git commit -m "작업 내용"
```

### 4. 팀 정책에 따라 merge 대신 rebase를 쓸 수도 있음

현재 정리한 방식은 **merge 기반 workflow**입니다.  
팀에서 commit 이력을 직선형으로 관리한다면:

```bash
git rebase origin/dev
```

를 사용할 수도 있습니다.  
다만 실무에서 이미 원격에 push한 feature 브랜치를 다른 사람과 공유 중이라면 rebase는 주의해야 합니다. 이 경우에는 merge가 더 안전합니다.

### 5. 삭제 전 확인 명령

로컬 feature 삭제 전 확인:

```bash
git branch
git status
git log --oneline --decorate -n 5
```

원격 feature 삭제 확인:

```bash
git branch -r
```

## 최종 판단

제시한 절차는 큰 방향은 맞지만, 실무용으로는 아래처럼 바꾸는 것이 더 좋습니다.

```text
git fetch --prune origin
→ git switch -c feature/user origin/dev
→ 작업
→ git status / git diff 확인
→ commit
→ git fetch origin
→ git merge origin/dev
→ 충돌 해결
→ 테스트
→ push
→ MR
→ Delete source branch
→ git fetch --prune origin
→ dev 최신화
→ local feature 삭제
```

가장 중요한 개선점은 **신규 브랜치를 로컬 dev가 아니라 `origin/dev`에서 바로 생성하는 것**, **merge 전에 작업을 commit 해두는 것**, **충돌 해결 후 반드시 테스트하는 것**, **로컬 feature 삭제는 MR merge 확인 후 진행하는 것**입니다.
